编写C语言程序时必须有#include这一行吗？为什么有时候不需要？

发表于 2024-11-26 分类于 编程基础 ， C语言

本文旨在解决 C 语言初学者对 #include 的常见困惑。由于市面上部分教材编排逻辑的缺失，导致很多初学者陷入了“学编程要先学 C，学 C 要先懂编程”的死循环。本文将从模块化设计的角度，重新审视 C 语言程序的结构。

前言：打破教材编排的“Bug”

市面上的 C 语言教程，往往存在一个严重的逻辑缺陷：无脑陈列细节，缺乏主线逻辑。

这种风气甚至可以追溯到 C 语言之父 Dennis Ritchie 的经典著作《C程序设计语言》。严格来说，这本书是一本“C 语言特性手册”，面向的是“初学 C 语言的程序员”，而非“编程零基础初学者”。然而，后世的许多教材（包括《C Primer Plus》等）都沿用了这种“字典式”的编排顺序，导致初学者在 Hello World 阶段就迷失了方向。

本文尝试打破这一惯例，通过第一课就讲清楚 #include，引出模块化设计的核心思想，帮助你建立从一行代码到完整项目的宏观认知。

第一课：#include 与模块化编程

程序即机器

一个优秀的程序设计，就像设计一台复杂的机器。它不是一块铁板，而是由多个功能模块组合而成。这些模块各司其职，最终由一个主体程序（通常是 main）统一调度。

让我们以编写一个“C 语言玩具车”程序为例，看看模块化思维是如何运作的。

1.0 版：单文件原型

在最原始的阶段，我们将所有功能模块写在一个文件中。这虽然简单，但已经体现了“模块化”的雏形——main 函数作为指挥官，负责调度各个功能函数。

  1. 如果你自己实现了所需功能：
    printf 并不是 C 语言语法的核心，它只是一个库函数。如果我自己写了一个打印函数，或者使用其他库，就不需要 stdio.h。

总结

  • #include <stdio.h> 不是仪式：它不是写 C 程序的“起手式”或魔法咒语。
  • 按需引入：你需要用什么功能，就包含对应的头文件。不用键盘操作，就不需要 <keyboard.h>（假设存在）；不用网络操作，就不需要 <network.h>。
  • 设计思维：学习 C 语言，不应从死记硬背 #include 开始，而应从理解“我要造一台什么样的机器”以及“这台机器由哪些模块组成”开始。

当你理解了模块化设计，#include 就不再是冰冷的代码行，而是你组装机器时拧下的一颗螺丝钉，用在哪里，就在哪里拧紧。
