蓝桥杯真题：幸运数字 (哈沙德数)

发表于 2024-04-08 更新于 2026-05-14 分类于 算法竞赛

题目描述

哈沙德数（Harshad Number）是指在某个固定的进位制当中，可以被各位数字之和整除的正整数。

幸运数字定义：如果一个整数在二进制、八进制、十进制、十六进制下均为哈沙德数，那么这个数字就是幸运数字。

前 10 个幸运数字的十进制表示为：1, 2, 4, 6, 8, 40, 48, 72, 120, 126…

问题：请问第 2023 个幸运数字是多少？你只需要输出这个整数的十进制表示即可。

解题思路

本题是一道典型的枚举模拟题。由于我们需要找到第 2023 个符合条件的数，且该位置的数据规模在整型范围内，直接暴力枚举并判断即可。

具体步骤如下：

  1. 遍历整数：从 1 开始遍历所有正整数。
  2. 进制转换：对于每个整数，分别计算其在二进制、八进制、十进制、十六进制下的字符串表示。
    • Python 中可直接使用内置函数 bin(), oct(), hex()，注意它们生成的字符串带有前缀（如 0b, 0o, 0x），切片时需要去掉。
  3. 数位求和：编写一个函数，计算某个进制字符串下的各位数字之和。
    • 注意十六进制中包含 a-f，对应十进制的 10-15，需特殊处理。
  4. 整除判断：判断当前整数是否能被这四种进制下的数位和整除。
  5. 计数与输出：如果同时满足四种进制的哈沙德数条件，则计数器加 1。当计数器达到 2023 时，输出该数字并结束程序。

代码实现

以下是优化后的 Python 代码实现：

```python
def check_harshad(n, num_str):
“””
判断数字 n 是否为特定进制字符串 num_str 下的哈沙德数
“””
digit_sum = 0
for char in num_str:

    # 处理十六进制中的字母
    if char == 'a':
        digit_sum += 10
    elif char == 'b':
        digit_sum += 11
    elif char == 'c':
        digit_sum += 12
    elif char == 'd':
        digit_sum += 13
    elif char == 'e':
        digit_sum += 14
    elif char == 'f':
        digit_sum += 15
    else:
        digit_sum += int(char)

# 避免除数为0的情况（虽然题目为正整数，但在二进制中 '0' 的情况需视具体数值而定）
# 实际上 n >= 1 时，二进制表示至少有一个 '1'，sum 不会为 0
if digit_sum == 0:
    return False

return n % digit_sum == 0

count = 0
i = 1

使用 while 循环进行无限遍历，直到找到目标

while True:

# 获取不同进制的字符串表示，并去除前缀 (0b, 0o, 0x)
bin_str = bin(i)[2:]
oct_str = oct(i)[2:]
dec_str = str(i)
hex_str = hex(i)[2:]

# 判断是否在四种进制下均为哈沙德数
# 注意：十进制传入 str(i)，即 dec_str
if (check_harshad(i, bin_str) and 
    check_harshad(i, oct_str) and 
    check_harshad(i, dec_str) and 
    check_harshad(i, hex_str)):

    count += 1

    if count == 2023:
        print(f"第 2023 个幸运数字是: {i}")
        break

i += 1

代码优化点说明

相比原始代码，优化后的版本做了以下改进：

  1. 函数命名与结构：将函数名改为 check_harshad，更具语义化；增加了详细的注释说明。
  2. 循环控制：将巨大的 range 改为 while True 循环，逻辑更加清晰自然，避免了魔法数字 10000000000000000。
  3. 字符串切片：显式地处理了 Python 进制转换函数产生的前缀（如 bin(i) 返回 '0b101'，需切片 [2:]），原代码中可能隐式包含了前缀字符导致计算错误（注：原题代码中似乎直接使用了转换后的字符串，如果包含前缀字符如 ‘b’ 或 ‘x’ 会导致数位和计算错误，优化后已修正）。
  4. 代码可读性：将条件判断拆分为多行，逻辑更清晰。
